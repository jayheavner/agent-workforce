# Dispatch subagent_type Guard — Combined Spec + Plan (Small Tier)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make it impossible for an orchestrator Agent dispatch to go out with a missing or invalid `subagent_type` and silently stall the task.

**Architecture:** Two enforcement layers. (1) An unmissable hard rule in the orchestrator definition that every Agent dispatch must set `subagent_type` to exactly one of the nine named specialists. (2) A deterministic PreToolUse(Agent) hook that runs in the orchestrator's own session and blocks any Agent call whose `subagent_type` is missing, empty, or not one of the nine specialists — including the harness default `general-purpose`.

**Tech Stack:** Bash + jq (same as the existing policy and cost hooks). No new dependencies.

## Background — root cause (plain sentences)

In a real orchestrator session, an Agent dispatch was issued without an explicit `subagent_type`. The Claude Code harness filled the field with its built-in default, `general-purpose`, which is not one of this team's nine specialists. The dispatch hard-failed with `Agent type 'general-purpose' not found`, no specialist ran, and the task silently stalled.

Why it was possible: the orchestrator definition (`agents/orchestrator.md`) registers only a **PostToolUse(Agent)** cost hook. There is no **PreToolUse** gate on the Agent tool, so nothing inspects a dispatch before it leaves. The existing policy hook (`hooks/agent-team-policy.sh`) is a *subagent-side* guard — it runs inside each specialist's own session keyed by ROLE and matches Bash/Write/Edit, never the orchestrator's Agent tool — so it structurally cannot see this failure. The dispatch guard must be a new orchestrator-side PreToolUse(Agent) hook.

## Enforcement-layer decision (one line)

Instruction **plus** deterministic guard — a silent stall on a forgotten field is a real reliability bug that instruction-only cannot make impossible, since an LLM can always forget a rule; the hook makes recurrence impossible regardless of orchestrator attention.

Why a **new** hook file rather than extending `agent-team-policy.sh`: that policy script is role-keyed and subagent-side (guards Bash/Write/Edit inside specialists); the dispatch guard is orchestrator-side, matches a different tool (Agent), and inspects a different field (`subagent_type`). Bolting them together would conflate two unrelated guard surfaces. A small dedicated hook matches the team's existing one-hook-per-concern convention (the cost hook is already its own file).

## Global Constraints

- Bash + jq only; no package installs (builder policy permanently forbids them).
- No `rm`/`mv`/shell deletion of files; the builder edits/overwrites in place via Edit/Write.
- Hook files under `hooks/` are shell scripts; the entry-point/executed hooks get `chmod +x` at install; this new hook is executed directly by the harness, so it must be executable.
- Config-safety discipline: validate every changed shell/JSON before it is considered installed (`bash -n`, `jq empty`), and the manifest is regenerated by `install.sh` — never hand-edited.
- Keep `agents/orchestrator.md` coherent; it is currently 119 lines and must stay well under the ~300-line ceiling.

## Task 1: Add the dispatch-guard hook file

**Files:**
- Create: `hooks/agent-team-dispatch-guard.sh`

**Interfaces:**
- Produces: a PreToolUse hook script invoked as `agent-team-dispatch-guard.sh` with hook JSON on stdin. Exit 0 = allow, exit 2 = block (stderr message returned to the orchestrator). Consumes `.tool_name` and `.tool_input.subagent_type` from the hook JSON.

- [ ] **Step 1: Create the hook file with this exact content**

```bash
#!/usr/bin/env bash
# agent-team-dispatch-guard.sh — PreToolUse(Agent) guard for the orchestrator.
# Blocks any Agent dispatch whose subagent_type is missing, empty, or not one
# of the nine named team specialists (so a forgotten field can never default to
# 'general-purpose' and stall the task silently).
# Hook JSON on stdin. Exit 0 = allow. Exit 2 = block (stderr returned to agent).
set -u

VALID=" architect builder verifier reviewer deployer researcher ops scribe ticketer "

INPUT="$(cat)"
TOOL="$(printf '%s' "$INPUT" | jq -r '.tool_name // empty')"

# Only police Agent dispatches; anything else passes through untouched.
[ "$TOOL" = "Agent" ] || exit 0

TYPE="$(printf '%s' "$INPUT" | jq -r '.tool_input.subagent_type // empty')"

if [ -z "$TYPE" ]; then
  printf 'agent-team dispatch guard: this Agent dispatch has no subagent_type. Every dispatch MUST set subagent_type to exactly one of: architect, builder, verifier, reviewer, deployer, researcher, ops, scribe, ticketer. Re-issue the dispatch with an explicit subagent_type.\n' >&2
  exit 2
fi

case "$VALID" in
  *" $TYPE "*) exit 0 ;;
  *)
    printf 'agent-team dispatch guard: subagent_type "%s" is not a team specialist. Use exactly one of: architect, builder, verifier, reviewer, deployer, researcher, ops, scribe, ticketer. (The harness default "general-purpose" is not a team agent and will hard-fail.)\n' "$TYPE" >&2
    exit 2
    ;;
esac
```

## Task 2: Add the test suite for the guard

**Files:**
- Create: `tests/test_dispatch_guard.sh`

**Interfaces:**
- Consumes: `hooks/agent-team-dispatch-guard.sh` from Task 1.
- Produces: an executable test script that exits 0 when all cases pass, nonzero otherwise — the shape `install.sh` uses for the other hook test suites.

- [ ] **Step 1: Write the failing test file with this exact content**

```bash
#!/usr/bin/env bash
# tests/test_dispatch_guard.sh — verifies the PreToolUse(Agent) dispatch guard.
set -u
HERE="$(cd "$(dirname "$0")" && pwd)"
GUARD="$HERE/../hooks/agent-team-dispatch-guard.sh"
PASS=0
FAIL=0
RC=0

run() { # $1 json
  set +e
  printf '%s' "$1" | bash "$GUARD" >/dev/null 2>&1
  RC=$?
  set -u
}

agent_json() { jq -cn --arg t "$1" '{tool_name:"Agent",tool_input:{subagent_type:$t}}'; }

expect() { # $1 expected_rc, $2 json, $3 label
  run "$2"
  if [ "$RC" -eq "$1" ]; then
    PASS=$((PASS+1))
  else
    FAIL=$((FAIL+1))
    echo "FAIL [$3]: expected=$1 got=$RC"
  fi
}
expect_allow() { expect 0 "$1" "$2"; }
expect_block() { expect 2 "$1" "$2"; }

# All nine valid specialists allow.
for a in architect builder verifier reviewer deployer researcher ops scribe ticketer; do
  expect_allow "$(agent_json "$a")" "valid: $a allows"
done

# Missing / empty / harness-default / unknown all block.
expect_block "$(jq -cn '{tool_name:"Agent",tool_input:{description:"do a thing"}}')" "missing subagent_type blocks"
expect_block "$(agent_json '')" "empty subagent_type blocks"
expect_block "$(agent_json 'general-purpose')" "general-purpose blocks"
expect_block "$(agent_json 'designer')" "unknown type blocks"

# Substring of a valid name must NOT match (space-delimited membership).
expect_block "$(agent_json 'archi')" "substring 'archi' blocks"
expect_block "$(agent_json 'build')" "substring 'build' blocks"

# A non-Agent tool passes through untouched.
expect_allow "$(jq -cn '{tool_name:"Bash",tool_input:{command:"ls"}}')" "non-Agent tool passes"

echo "dispatch-guard tests: PASS=$PASS FAIL=$FAIL"
[ "$FAIL" -eq 0 ]
```
